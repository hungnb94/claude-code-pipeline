# ADR 0007: Hook-driven state advancement instead of agent-composed advance scripts

**Status**: Accepted
**Date**: 2026-07-02

## Context

Every step advance was driven by a `python3 -c "..."` snippet generated by `pipeline_utils.js` and handed to Claude to run via Bash between steps. This gave Claude, not the hook, control over the actual state mutation: the snippet was plain text in a chat turn, so Claude could edit it before running it (redirect `current_step` to any step in the graph, append arbitrary entries to `completed_steps`, or fabricate a shell step's pass/fail) or bypass it entirely (`Edit`/`Write`/`cat >`/`jq`/`sed` directly on the session file — nothing guarded it). For shell steps specifically, Claude ran the real commands itself and then chose which of the two pre-rendered snippets (success or fail) to run, so a fabricated "pass" required no code changes at all, just running the other snippet.

An alternative — leaving the trust model as-is and only translating the interview prompt — was rejected because it does not address the underlying request: preventing Claude from freely choosing how the pipeline advances.

## Decision

Two mechanisms replace agent-composed state mutation:

1. **Shell steps run inside the `Stop` hook itself.** `check_pipeline.js` executes `step.commands` via `child_process` when it encounters a shell step, captures stdout/stderr into `shared_state[<step>_output]`, and picks `next` or `next_fail` from the real exit code — chained through consecutive shell steps in one hook invocation until a non-shell step or a failure is reached. Claude is never asked to run or report on these commands.
2. **Agent/interview steps advance via a fixed CLI**, `hooks/pipeline_advance.js`, replacing the freeform snippet. Claude supplies `--session <id>`, `--step <name>` (rejected if it doesn't match `state.current_step`), and `--output <text>` (agent) or `--requirements <text>` (interview entry step). The script looks up `next` from `pipeline.yaml` itself — Claude cannot specify a destination step.

A new PreToolUse guard (`guard_state.js`) denies `Edit`/`Write`/`MultiEdit` on `.pipeline/sessions/**/*.json` and `.pipeline/state.json` unconditionally, and denies `Bash` commands that reference those paths unless the command also references `pipeline_advance.js`.

## Consequences

- Shell step outcomes are now ground truth, not agent-reported — this also fixes `{{verify_output}}` / `{{validate_plugin_output}}` in `pipeline.yaml`, which referenced shared-state keys that were never populated under the old scheme.
- The `Stop` hook's registered timeout must be raised (`hooks.json`), since one invocation may now run several real command sets (e.g. `verify` → `lint` → `jscpd_check` → `validate_plugin`) synchronously before yielding back to Claude.
- The Bash half of the file guard is a best-effort substring check, not a sandbox — a sufficiently adversarial agent with full shell access can still evade it (e.g. writing the target path through indirection the guard doesn't pattern-match). It closes the direct, low-effort tampering paths (Edit tool, `cat >`, `jq`, `sed`, naming the file literally in a one-liner); it is not a hard security boundary.
- Claude still self-reports _when_ an agent/interview step is done and _what_ the summary text is — that judgment is inherent to LLM-driven steps and is not eliminated, only the ability to choose the destination step or fabricate shell results.
